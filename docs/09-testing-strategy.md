# 测试设计策略

## 文档目的

这份文档定义 ToyLink AI MVP 的整体测试策略。

它主要回答下面几个问题：

- 我们优先测什么
- 哪些层应该怎么测
- 什么时候用 Fake，什么时候才需要 Mock
- 集成测试要做到哪一层才算有价值

这份文档不是测试代码本身，而是后续测试实现的设计依据。

如果你是初级开发者，可以把它理解成：

“不是先想到什么就测什么，而是先保护最容易出问题、出问题代价最大的地方。”

## 测试总原则

ToyLink AI 的 MVP 测试必须遵守以下原则：

- 优先保护高风险核心逻辑，而不是追求测试数量
- 优先测试“系统有没有做对决策”，再测试“界面有没有显示对”
- 协议编码、安全规则、错误映射、MCP 路由是最高优先级
- BLE 测试尽量隔离插件层，不直接依赖 `flutter_blue_plus` 的真实行为
- `Failure` 映射必须稳定
- `stopAll()` 的优先级必须有测试保护
- EMS 安全限制必须有单独测试保护

一句话总结：

先测“容易让系统出大事的逻辑”，再测“看起来完整但价值没那么高的部分”。

## 为什么采用“核心逻辑优先”策略

ToyLink AI 不是普通的信息展示 App，它有几个明显的高风险点：

- BLE 协议命令错误会直接影响硬件控制
- EMS 规则如果出错，安全风险高
- MCP 如果路由错误，AI 可能控制错设备或在错误时机调用
- `Failure` 如果不统一，UI、日志和 MCP 会各说各话

所以我们的 MVP 测试策略不追求平均铺开，而是先保护这些最关键的链路。

这对初级开发者特别重要，因为一开始最容易陷入一个误区：

“我写了很多测试，所以项目很安全。”

实际上，如果测的都是低价值页面展示，而没有保护协议、安全和路由核心，那测试数量再多也不等于安全。

## 测试分层说明

ToyLink AI MVP 推荐采用三层测试结构：

1. 单元测试
2. Widget 测试
3. 集成测试

### 单元测试

作用：

- 保护最核心、最稳定、最应该长期可靠的业务规则

### Widget 测试

作用：

- 保护高价值交互页面在关键状态下的正确展示和交互反馈

### 集成测试

作用：

- 验证关键主链路在多个模块协作时能不能真正闭环

## 单元测试范围

单元测试是本项目 MVP 测试体系的重心。

### 优先覆盖对象

- `SosexyProtocolCodec`
- `SafetyGuard`
- `Failure` 映射器
- `ActiveDeviceRegistry`
- application 层 use case
- `McpToolRouter`
- 存储抽象层
- 设备模板解析逻辑
- 协议配置读取逻辑

### 为什么优先测这些

因为这些对象决定的是“系统有没有做对关键决策”：

- 命令编码对不对
- 安全拦截对不对
- 错误分类对不对
- 控制目标路由对不对
- MCP 工具调用有没有真正走统一链路

### 必测场景

#### 协议与编码

- `SosexyProtocolCodec` 是否输出正确字节结构
- suck、vibe、ems、stop 命令是否按规格构建
- 非法模式值是否被拒绝
- 非法强度值是否被拒绝

#### 安全规则

- EMS `0..8` 是否允许直接通过
- EMS `9..20` 是否触发确认要求
- EMS `>20` 是否被直接拒绝
- `set_all` 中包含危险 EMS 值时是否整次拒绝

#### 路由与状态

- 无活跃设备时是否统一得到 `Failure.noActiveDevice`
- 活跃设备切换后是否只路由到最新设备
- `stopAll()` 是否具有更高优先级

#### 错误映射

- BLE 写入失败是否映射到 `Failure.deviceWrite`
- 权限拒绝是否映射到 `Failure.permissionDenied`
- 协议不匹配是否映射到 `Failure.protocolUnsupported`
- 未知异常是否最终收敛到 `Failure.unknown`

#### 存储层

- 普通设置是否进入正确存储层
- 敏感数据是否没有写入非安全存储
- 设备模板和协议配置是否能正确读写

## Widget 测试范围

因为本项目采用“核心逻辑优先”，所以 Widget 测试只覆盖高价值页面和关键状态，不追求平均铺满。

### 优先覆盖页面

- 扫描连接页
- 手动控制页
- 首页 MCP 主卡片
- App Lock 页面
- 关键失败提示态

### 为什么优先测这些页面

因为这些页面最直接影响用户主路径：

- 找设备
- 连设备
- 控设备
- 看 MCP 状态
- 处理安全与失败提示

### 必测交互

#### 扫描连接页

- 扫描中状态是否正确显示
- 无结果时是否显示空状态
- 权限被拒绝时是否显示正确引导
- 连接中时是否显示 loading 状态

#### 手动控制页

- 无活跃设备时是否引导用户去扫描页
- EMS 超软上限时是否显示确认或拦截 UI
- `stopAll()` 按钮是否始终可见且位置明确
- 错误态是否基于 `Failure` 正确渲染

#### 首页 MCP 主卡片

- `mcpStopped` 时是否显示启动引导
- `mcpRunning` 时是否展示服务状态
- `runningWithoutDevice` 场景是否有明确说明

#### App Lock 页面

- 锁定态是否阻止敏感页面继续使用
- 解锁成功后是否能回到原目标页

## 集成测试范围

MVP 集成测试必须至少做到“本地 MCP 调用闭环”。

这不是可选增强项，而是本项目区别于普通 BLE App 的核心能力验证。

### 闭环 1：扫描 -> 连接 -> 控制

目标：

- 验证扫描、连接、活跃设备建立和一次控制命令的完整主路径

建议流程：

1. 启动扫描流程
2. 返回候选设备列表
3. 选中一个设备
4. 建立活跃设备
5. 进入控制页可用状态
6. 发出一次本地控制命令
7. 状态成功回流

### 闭环 2：MCP tool -> application -> active device

目标：

- 验证本地 MCP 工具调用真正穿过统一业务链路

建议流程：

1. 启动本地 MCP 服务
2. 注册工具
3. 调用 `set_suck`
4. 请求进入 `McpToolRouter`
5. 命中 `SafetyGuard`
6. 路由到 `ActiveDeviceRegistry`
7. `FakeToyDevice` 收到命令
8. 返回结构化状态

### 闭环 3：MCP 运行中 + 设备断开

目标：

- 验证设备断开后 MCP transport 本身不崩溃

建议流程：

1. 启动 MCP 服务
2. 让活跃设备进入断开状态
3. 调用 `get_status` 或控制 tool
4. 返回结构化错误
5. MCP 服务仍保持运行状态

### 闭环 4：EMS 超软上限

目标：

- 验证安全规则不会在 MCP 调用链路里被绕过

建议流程：

1. 调用 `set_ems(intensity > 8)`
2. 请求进入 `SafetyGuard`
3. 返回 `ems_confirmation_required`
4. 不允许静默透传到设备

## Fake / Mock 策略

MVP 默认采用：

- `FakeHardwareRepository`
- `FakeToyDevice`
- `FakeMcpService`
- `FakeChatProvider`
- 少量 `Mocktail` 用于边缘依赖

## 为什么优先用 Fake

### 1. Fake 更稳定

Fake 更适合描述“一个有状态的系统对象会如何响应”，而不是只描述“某个方法被调了几次”。

### 2. Fake 更适合状态流测试

ToyLink AI 有大量状态流：

- 扫描状态
- 连接状态
- MCP 状态
- 设备状态

Fake 更容易模拟这种连续变化。

### 3. Fake 更适合初级开发者理解

对于初级开发者来说：

- Fake 更像“一个假的但完整的小系统”
- Mock 更像“只为了断言调用而拼出来的测试道具”

前者更容易理解，也更适合维护。

## 什么时候才考虑 Mock

只有以下情况才优先考虑 Mock：

- 很难用 Fake 覆盖的边缘依赖
- 需要验证交互次数或特定调用序列
- 外部接口本身非常薄，Fake 成本高于 Mock

默认原则：

- 不直接 Mock `flutter_blue_plus` 插件方法作为主策略
- 不把插件行为细节带入大多数业务测试

## 不建议现在做的测试

为了避免把时间花在低价值方向，MVP 阶段不建议优先投入：

- 大量 Golden 测试
- 真实 BLE 硬件自动化回归作为主测试手段
- 直接依赖真实 Claude 或中转服务的测试
- 过度细化的插件层 Mock 测试
- 只为覆盖率数字而写的低价值测试

### 为什么不建议

因为这些测试要么：

- 太脆弱
- 太依赖外部环境
- 太难维护
- 对 MVP 当前阶段的核心风险保护不够强

## MVP 最小测试闭环

如果只验证 MVP 是否具备最基本的测试安全网，至少应具备下面这些：

### 单元测试最小闭环

- `SosexyProtocolCodec`
- `SafetyGuard`
- `Failure` 映射
- `ActiveDeviceRegistry`
- `McpToolRouter`

### Widget 测试最小闭环

- 扫描连接页
- 手动控制页中的 EMS 提示
- 首页 MCP 主卡片状态

### 集成测试最小闭环

- 扫描到连接到控制
- MCP tool 调用到活跃设备
- EMS 超限拦截

## 验收标准

一份合格的 MVP 测试体系，至少应满足：

- 高风险逻辑都有测试保护
- `Failure` 语义在 UI 和 MCP 中保持一致
- BLE 相关业务测试主要依赖 Fake，而不是插件直连 Mock
- MCP 集成测试已规划到闭环级别
- `stopAll()` 和 EMS 安全规则有明确测试
- 测试设计和现有文档不冲突

## 给初级开发者的建议

如果你不知道先写哪个测试，可以按下面顺序开始：

1. 先写 `SafetyGuard`
2. 再写 `Failure` 映射
3. 再写 `SosexyProtocolCodec`
4. 再写 `McpToolRouter`
5. 最后再补页面级 Widget 测试

这样做的好处是：

- 先把系统最容易出问题的规则保护住
- 后面写 UI 时更安心
- 不会一开始就陷入低价值测试工作

## 总结

ToyLink AI 的测试策略不是“尽可能多写测试”，而是“尽可能优先保护最危险的逻辑”。

对于这个项目来说，最危险的地方不是按钮有没有摆对，而是：

- 协议对不对
- 安全规则有没有漏
- MCP 有没有走错路
- 错误是不是被错误地解释了

把这些守住，MVP 才算真正有测试价值。
