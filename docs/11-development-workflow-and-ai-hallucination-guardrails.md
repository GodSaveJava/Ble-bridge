# 开发工作流与 AI 幻觉防护

## 文档目的

这份文档把 ToyLink AI 的日常开发方式正式落地，目标是让团队成员和 AI 助手都按同一套流程推进工作。

它主要解决两个问题：

- 每次开发任务应该怎么从需求走到实现
- 在使用 AI 辅助开发时，如何避免“说得很像对的，但其实是猜的”

如果你是初级开发者，可以把这份文档理解成：

“不是想到什么就开始写，而是按固定节奏推进，并且对 AI 的输出保持验证意识。”

## 为什么这份文档很重要

ToyLink AI 不是普通 CRUD 项目，它同时涉及：

- BLE 硬件控制
- 协议编码
- MCP 工具调用
- EMS 安全边界
- 本地安全与隐私

在这种项目里，最危险的情况不是“代码还没写完”，而是：

- 以为自己理解了协议，其实理解错了
- 以为 AI 给的插件 API 是真的，其实是编出来的
- 以为某个状态已经覆盖，其实少了断开或超限场景
- 以为 MCP 和 UI 走的是同一套逻辑，其实已经分叉

所以我们必须把开发流程和 AI 风险防护一起制度化。

## 引入 Context-First 原则

参考 `context-hub` 这类守则后，团队应正式采用一条默认原则：

- 先拿可靠上下文，再做设计和编码

这里的“可靠上下文”指的是：

- 当前仓库里的正式文档
- 官方包文档
- 已验证的协议笔记
- 当前代码库中的真实实现

而不是：

- 记忆里的 API
- 搜索结果里看起来像答案的片段
- AI 自己补全出来的“合理做法”

## Context-First 工作方式

每次进入不确定任务时，默认先走这个循环：

1. 搜索可信来源
2. 获取聚焦资料
3. 基于已验证资料实现
4. 记录缺口、假设和修正

这条规则的价值在于：

- 降低插件 API 幻觉
- 降低 BLE 协议猜错的风险
- 降低“不同人重复踩同一个坑”的概率

## 什么叫“聚焦资料”

不是每次都去读一整套大文档，而是先拿最小但足够可靠的资料集。

例如：

- 做 `flutter_blue_plus` 连接流程时，优先拿连接、扫描、服务发现相关文档
- 做 MCP tool 时，优先拿本地 MCP 合同和 transport 相关资料
- 做 SOSEXY 编码时，优先拿协议规格、通道映射和已验证样例

这样可以减少噪音，也更适合 AI 和开发者协作。

## 发现资料缺口时应该怎么做

不要这样做：

- 觉得“差不多应该是这样”就直接写死

要这样做：

- 明确标记这是缺口
- 区分“已验证”和“推断”
- 把结论补进仓库文档，而不是留在聊天记录里

推荐处理方式：

- 写入相关专题文档
- 加入 `Open Questions`
- 加入实现阶段注意事项
- 如果是通用经验，补进 `CODEX.md` 或本文件

## 开发总模式

ToyLink AI 采用以下模式：

- 文档驱动
- 分阶段迭代
- 小步实现
- 小步验证
- 完成后回看架构

一句话总结：

先确认边界，再实现小闭环，再验证，再继续。

## 标准开发流程

每次开发任务都应尽量按下面 8 步推进。

### 1. 先看已有文档

开发前先检查相关文档，而不是直接开写。

优先查看：

- `CODEX.md`
- `docs/06-implementation-roadmap.md`
- `docs/07-failure-catalog.md`
- 当前任务直接相关的专题文档

例如：

- 做页面前看 `08-page-and-interaction-flow.md`
- 做 provider 前看 `10-routing-and-provider-map.md`
- 做测试前看 `09-testing-strategy.md`

### 2. 明确本次任务边界

每次任务必须回答清楚：

- 本次只做哪个 feature 或 layer
- 本次不做什么
- 输入是什么
- 输出是什么
- 完成标准是什么

示例：

- 本次只实现 `core + domain`
- 不接 BLE、不做页面、不做 MCP

### 3. 先验证“事实”，再做设计或编码

在写代码前先确认这些事实：

- 当前仓库已有代码结构
- 插件 API 是否真实存在
- 相关文档是否已有明确约束
- 需要依赖的协议细节是否已验证
- 是否已经拿到最小可靠资料集

这一步是防止 AI 幻觉的第一道墙。

### 3.1 优先使用可追溯来源

如果一个实现依赖外部知识，优先级应为：

1. 当前仓库正式文档
2. 官方文档或官方源码
3. 已验证协议记录
4. 可信社区资料
5. AI 生成解释

AI 生成解释只能作为辅助，不应高于正式来源。

### 4. 先设计最小实现路径

不要一开始设计一个过大的方案。

每次都优先问：

- 这次最小闭环是什么
- 要先接通哪条主路径
- 哪些增强能力可以晚一点做

比如：

- 先让 `ToyDevice` 和 `Failure` 成型
- 再让 `ActiveDeviceRegistry` 能跑通
- 再让 `SosexyDevice` 能接一条最小命令链路

### 5. 小步实现

每次只实现一小块，而且尽量让它是可验证的。

正确示例：

- 先实现 `SafetyGuard`
- 再实现 `Failure` 映射
- 再实现 `SosexyProtocolCodec`

错误示例：

- 一次性把扫描页、控制页、BLE、MCP 一起写完

### 6. 小步验证

每实现一小块，都要立刻验证：

- 逻辑是否符合文档
- 是否破坏了架构边界
- 是否需要测试补充
- 是否存在不确定实现

### 7. 架构回看

完成一个小阶段后，主动回看：

- 是否绕过了 `ToyDevice`
- 是否绕过了 `SafetyGuard`
- 是否让 UI 直接接触 infrastructure
- 是否让页面拥有了系统真相状态
- 是否让错误处理脱离 `Failure`

### 8. 更新必要文档

如果实现中发现：

- 原设计缺了关键细节
- 原设计有误
- 实际实现需要新的约束

就要更新相关文档，而不是只把“真实情况”藏在代码里。

### 8.1 把新知识沉淀下来

凡是以下内容，一旦确认有效，就应该落回仓库：

- 插件使用限制
- BLE 协议补充结论
- MCP 调用边界
- 测试替身经验
- 已验证 workaround

不要让团队每次都重新问 AI 或重新踩坑。

## 推荐开发节奏

团队推荐按以下节奏推进：

1. 先完成一个清晰小目标
2. 做最小测试验证
3. 做一次自查
4. 再进入下一小目标

不要连续写很多天代码后才第一次回看架构。

## 任务拆分规则

每个任务都应该尽量满足以下条件：

- 可以在单个主题内闭环
- 不同时跨太多层
- 完成后能被验证
- 不需要临时发明新规则

推荐拆分方式：

- 按 layer 拆
- 按能力拆
- 按主链路拆

不推荐拆分方式：

- 纯按页面视觉拆
- 纯按“谁有空谁写什么”拆
- 一次性做太多不相关内容

## 每次开发前的检查清单

开始编码前，先检查：

- 我现在做的是哪个阶段
- 相关文档我有没有看
- 本次任务是否只聚焦一个边界
- 本次任务的完成标准是否明确
- 我依赖的协议/API/状态机是不是已验证

## 每次开发后的检查清单

完成后，检查：

- 是否符合 `CODEX.md`
- 是否符合当前专题文档
- 是否绕过了 `ToyDevice`
- 是否绕过了 `Failure`
- 是否绕过了 `SafetyGuard`
- 是否增加了没有记录的隐含规则
- 是否需要补测试
- 是否需要修正文档

## AI 幻觉是什么

在这个项目里，AI 幻觉通常不是“胡言乱语”，而是这些更危险的形式：

- 插件 API 名字看起来很合理，但实际不存在
- BLE 协议字节格式看起来像真的，但其实是猜的
- 某个 provider 或 route 结构听起来顺，但和现有文档冲突
- 某个测试结论被写成“已经验证”，其实从未跑过
- 把“推断”说成“事实”

最危险的一点是：

AI 幻觉往往写得非常自信。

## 在 ToyLink AI 中最常见的 AI 幻觉风险

### 1. 幻觉插件 API

例子：

- 编造 `flutter_blue_plus` 的不存在方法
- 假设某个前台服务插件有某种 callback

防护方式：

- 先看官方文档或当前依赖代码
- 不凭记忆写插件调用
- 优先基于最小可靠资料集，而不是模糊搜索结果

### 2. 幻觉 BLE 协议

例子：

- 自己补齐一个“看起来合理”的 SOSEXY 字节数组
- 未验证就假设 `setAll` 有原生组合命令

防护方式：

- 回到 `01-sosexy-protocol-spec.md`
- 未验证字段必须标记为推断
- 协议编码前先列出已知、未知和假设
- 一旦确认新结论，立刻补回文档

### 3. 幻觉架构一致性

例子：

- 以为某个页面 controller 只是轻量封装，实际上已经偷偷做了业务逻辑中心
- 以为 MCP 和 UI 共用逻辑，实际上已经各写了一套

防护方式：

- 对照 `CODEX.md`
- 对照 `02-domain-interfaces-and-state-machines.md`
- 对照 `10-routing-and-provider-map.md`

### 4. 幻觉测试覆盖

例子：

- 说“这个已经有测试保护了”，但其实只测了 happy path
- 说“已经验证 EMS 超限逻辑”，但只写了 UI 提示测试，没有测 `SafetyGuard`

防护方式：

- 对照 `09-testing-strategy.md`
- 明确是单元测试、Widget 测试还是集成测试
- 不夸大测试范围
- 不把“写了测试计划”当成“已经验证通过”

### 5. 幻觉实现完成度

例子：

- 功能 UI 做出来了，就说功能完成
- 路由能跳了，就说页面流程完成

防护方式：

- 回到 `06-implementation-roadmap.md`
- 对照阶段完成标准

## AI 输出的验证规则

后续使用 AI 参与开发时，默认必须按下面规则验证 AI 输出。

### 可以直接信任的内容

只限于：

- 已在当前仓库文档中明确写死的规则
- 已在当前代码中明确存在的结构
- 已经通过实际测试验证过的结果
- 来自官方或已确认来源的可追溯资料

### 不能直接信任的内容

- AI 回忆出来的插件 API
- AI 补全的协议细节
- AI 自己总结的“应该如此”
- 没有证据的测试结论
- 没有跑过的“已经实现”

### 默认动作

只要出现以下情况，就必须二次验证：

- 涉及 BLE 协议
- 涉及 EMS 安全
- 涉及 MCP 路由
- 涉及权限与前台服务
- 涉及插件调用
- 涉及错误映射
- 涉及外部知识迁移到本仓库规则

## AI 输出验证清单

每次使用 AI 生成方案、代码或解释后，至少检查：

1. 它说的 API 在当前依赖里真的存在吗？
2. 它说的协议字段在文档里真的被确认了吗？
3. 它有没有把“推断”说成“事实”？
4. 它有没有偷偷绕过已有架构边界？
5. 它有没有遗漏 `Failure`、`SafetyGuard`、`ToyDevice` 这几个核心对象？
6. 它说“测过”或“支持”，有证据吗？
7. 它依赖的外部资料是否可追溯到可信来源？
8. 这次确认出来的新知识，是否需要补回仓库文档？

## 什么时候必须停下来重新确认

出现以下任一情况，就不要继续往下写代码，而要先停下来确认：

- 协议细节和文档对不上
- 插件调用和官方文档对不上
- 页面状态和状态机文档对不上
- MCP 行为和工具合同对不上
- 实现为了省事开始绕过 `SafetyGuard` 或 `ToyDevice`
- 某个地方只能靠“我猜大概是这样”

## 团队协作建议

如果是多人或“人 + AI”协作，建议统一约定：

- 任何人修改高风险逻辑前先看相关文档
- 对协议、安全、MCP 的改动要写清楚依据
- 任务讨论里把“已验证”和“待确认”分开写
- 代码审查时优先看边界、安全和错误处理，而不是先看样式

## 给初级开发者的建议

你完全不需要因为 AI 说得很像样就立刻相信它。

更稳的做法是：

1. 先把 AI 当成加速器，不当成事实源
2. 涉及插件、协议、安全时一定回文档
3. 不确定就标记假设，不要偷偷写死
4. 优先问“这个有证据吗”，而不是“这个听起来合理吗”

这不是不信任 AI，而是专业开发必须有的习惯。

## 总结

ToyLink AI 后续开发应该采用：

- 文档驱动
- 分阶段迭代
- 小步实现
- 小步验证
- 完成后回看架构

同时必须默认警惕 AI 幻觉，尤其是在：

- 插件 API
- BLE 协议
- EMS 安全
- MCP 路由
- 测试覆盖
- 完成度判断

真正专业的开发团队，不是“完全不用 AI”，而是“会用 AI，但不会把 AI 的自信当成事实”。
