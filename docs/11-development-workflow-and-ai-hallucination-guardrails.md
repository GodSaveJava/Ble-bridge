# 开发工作流与 AI 幻觉防护

## 1. 文档目的

本文件规定 ToyLink AI 的日常开发方式，以及团队在使用 AI 辅助时必须遵守的验证规则。

它要解决两个问题：

1. 每次开发任务应该按什么顺序推进
2. 如何防止 AI 把“猜测”说成“事实”

ToyLink AI 涉及 BLE、MCP、EMS 安全、后台保活和隐私保护。这里最危险的不是“写得慢”，而是“看起来像对的，但实际上没有被验证”。

## 2. 默认开发模式

ToyLink AI 采用以下默认开发模式：

- 文档驱动
- 分阶段推进
- 小步实现
- 小步验证
- 完成后回看架构

一句话概括：

先拿到可靠上下文，再做设计；先跑通最小闭环，再逐步扩展。

## 3. Context-First 原则

在设计或实现任何不确定行为前，先拿可靠上下文。

可靠上下文包括：

- 仓库中的正式文档
- 当前代码中的真实实现
- 官方依赖文档
- 已验证的协议记录

不可靠上下文包括：

- 记忆中的 API
- 搜索结果里像答案的片段
- AI 自动补全出来的“合理推断”

默认顺序：

1. 查仓库文档
2. 查当前代码
3. 查官方资料
4. 标记仍未验证的部分
5. 再设计和实现

## 4. 标准任务流程

每次开发任务都尽量按这 8 步推进。

### 4.1 先看文档

优先检查：

- `CODEX.md`
- 当前阶段路线图
- 当前任务相关专题文档
- `docs/07-failure-catalog.md`

### 4.2 明确边界

开始前要回答清楚：

- 这次只做哪个 layer 或 feature
- 这次明确不做什么
- 输入是什么
- 输出是什么
- 完成标准是什么

### 4.3 先验证事实

实现前先确认：

- 依赖 API 是否真实存在
- 协议信息是否已验证
- 文档与代码是否一致
- 是否已有同类实现或约束

### 4.4 设计最小闭环

优先问：

- 这次最小闭环是什么
- 哪条主链路最值得先跑通
- 哪些增强能力可以延后

### 4.5 小步实现

不要一次性跨太多层。

推荐节奏：

- 先做领域模型
- 再做应用服务
- 再接基础设施
- 最后做页面和交互

### 4.6 小步验证

每完成一小块，都要检查：

- 逻辑是否符合文档
- 是否破坏架构边界
- 是否需要补测试
- 是否引入新的未验证假设

### 4.7 架构回看

完成小阶段后主动检查：

- 是否绕过了 `ToyDevice`
- 是否绕过了 `SafetyGuard`
- 是否让 UI 直接依赖 infrastructure
- 是否让页面持有系统真相状态
- 是否把错误处理绕开 `Failure`

### 4.8 反哺文档

如果实现中发现：

- 原设计缺关键细节
- 原设计有误
- 实际实现需要新增约束

就必须更新文档，而不是只把事实藏在代码里。

## 5. AI 幻觉高风险区

在 ToyLink AI 中，AI 最容易出错的地方有 5 类。

### 5.1 插件 API 幻觉

常见问题：

- 说某个 `flutter_blue_plus` 方法存在，但实际没有
- 假设某个前台服务插件有某个回调

防护方式：

- 先看官方文档
- 再看仓库当前依赖版本
- 不凭记忆写插件调用

### 5.2 BLE 协议幻觉

常见问题：

- 凭感觉补字节格式
- 把推测写成已验证协议

防护方式：

- 回到协议文档
- 明确区分 `verified` 和 `inferred`
- 任何新结论都要写回文档

### 5.3 架构一致性幻觉

常见问题：

- 以为 MCP 和 UI 共享逻辑，实际上各写了一套
- 以为 page controller 很轻，实际上偷偷承载了业务中心

防护方式：

- 对照 `CODEX.md`
- 对照状态机与 provider 映射文档
- 做完后检查依赖方向

### 5.4 测试覆盖幻觉

常见问题：

- 说“已经有测试保护”，其实只测了 happy path
- 说“已经验证过”，其实只是写了计划

防护方式：

- 区分单元、Widget、集成测试
- 明确说明哪些真的跑过
- 不夸大验证范围

### 5.5 完成度幻觉

常见问题：

- UI 做出来了就说功能完成
- 页面能跳转了就说流程完成

防护方式：

- 回到路线图里的完成标准
- 回到测试策略和验收标准
- 看闭环是否真的打通

## 6. AI 输出验证清单

每次使用 AI 输出方案、代码或分析后，都至少检查：

1. 提到的 API 在当前依赖里真实存在吗
2. 提到的协议字段在文档里真实确认了吗
3. 有没有把推测说成事实
4. 有没有绕过 `ToyDevice`、`SafetyGuard` 或 `Failure`
5. 说“已测试”时，是否真的有证据
6. 引用的外部资料是否可信、可追溯
7. 是否有新知识需要补回仓库文档

## 7. 必须停下来重确认的情况

出现以下任一情况，先不要继续写代码：

- 协议细节和文档对不上
- 插件调用和官方文档对不上
- 页面状态和状态机对不上
- MCP 行为和工具合同对不上
- 为了省事开始绕过 `SafetyGuard` 或 `ToyDevice`
- 某个关键点只能靠“我猜应该是这样”

## 8. 团队协作规则

多人或“人 + AI”协作时，统一遵守：

- 高风险改动前先看相关文档
- 协议、安全、MCP 的修改要写清依据
- 讨论里把“已验证”和“待确认”分开写
- 代码审查优先看边界、安全、错误处理

## 9. 给初级开发者的建议

把 AI 当成加速器，不要当事实源。

更稳的做法是：

1. 先问“这件事有证据吗”
2. 先看仓库文档，再看 AI 说法
3. 不确定就标记假设，不要偷偷写死
4. 学会分辨“能运行”和“真正确认”

## 10. 总结

ToyLink AI 后续开发必须坚持：

- 文档驱动
- 上下文优先
- 小步闭环
- 明确验证
- 及时回写文档

真正专业的协作方式，不是不用 AI，而是会用 AI，同时不把 AI 的自信当成事实。
