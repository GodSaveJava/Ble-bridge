# Claude Connector 接入向导设计

## 1. 文档目的

本文档定义 ToyLink AI 首发版本的 **Claude Connector 接入向导**。

它主要回答以下问题：

1. 用户在什么时机进入接入向导。
2. 首次接入 Claude 时，用户要完成哪些步骤。
3. 每一步页面应该展示什么、让用户做什么。
4. 如果配置失败，应该如何提示和回退。

本文档服务于：

- 产品页面设计
- 路由与状态设计
- 后续教程页实现
- 首次接入体验优化

本文档应与以下文档保持一致：

- `docs/16-product-goals-and-next-design-plan.md`
- `docs/19-claude-remote-mcp-architecture.md`
- `docs/07-failure-catalog.md`
- `CODEX.md`

## 2. 设计目标

这份向导不是普通帮助文档，而是首发主流程的一部分。

它的目标不是“给高级用户说明技术原理”，而是让用户在第一次使用时，能够完成以下结果：

1. 确认当前玩具和适配器已经就绪
2. 理解为什么要给 Claude 添加 connector
3. 拿到需要填入 Claude 的信息
4. 按步骤完成一次 connector 配置
5. 回到原来的 Claude 对话验证控制是否已经生效

一句话概括：

**让用户第一次接入 Claude 时不靠猜，不靠搜索，不靠外部教程。**

## 3. 设计原则

### 3.1 面向普通用户，不面向开发者

页面必须尽量使用中文自然语言，不应该默认把用户当作熟悉：

- MCP
- transport
- connector schema
- token header
- remote bridge

这些词可以出现，但必须伴随中文解释。

### 3.2 向导只做“接入”，不做“教育全世界”

向导只解决当前用户第一次把 Claude 接到 ToyLink 这件事。

它不承担：

- 全量 MCP 教学
- 多平台兼容说明
- Bridge 架构深度解释

### 3.3 每一步都必须可回退

用户可能在任何一步失败，所以每一步都必须提供：

- 当前状态
- 下一步
- 返回上一环节
- 去排查入口

### 3.4 向导要和系统真实状态绑定

向导不能只是一组静态说明。

它必须读真实状态，例如：

- 设备是否已连接
- 是否已绑定适配器
- 是否已验证
- Bridge 是否已就绪
- Connector 信息是否已生成

## 4. 向导入口时机

### 4.1 推荐主入口

推荐主入口放在：

- 低强度验证通过后的成功页
- MCP / Bridge 状态页中的主按钮
- 首页的“下一步建议”区域

### 4.2 自动引导规则

如果用户满足以下条件：

- 已连接设备
- 已绑定适配器
- 已完成验证
- 尚未完成 Claude 接入

系统应主动推荐进入：

`Claude 接入向导`

### 4.3 不允许直接进入完成步骤

如果用户尚未完成前置条件，向导不应允许直接跳到“复制 connector 信息”。

例如：

- 设备未连接
- 适配器未绑定
- 验证未完成

都应该先拦住，并给出返回路径。

## 5. 向导总流程

首发建议将向导设计为 5 步：

1. `环境检查`
2. `桥接准备`
3. `Claude 添加 connector`
4. `原对话验证`
5. `完成与故障排查`

总流程如下：

`检查本地就绪 -> 生成接入信息 -> 去 Claude 添加 connector -> 回原对话测试 -> 完成或排查`

## 6. 第一步：环境检查

### 6.1 目标

确认当前设备和本地控制环境是否满足 Claude 接入前置条件。

### 6.2 页面应展示的信息

- 当前设备状态
- 当前适配器状态
- 当前验证状态
- 当前后台保活状态
- 当前桥接状态

### 6.3 页面文案重点

建议用用户能理解的话表达：

- `玩具已连接`
- `当前模板已绑定`
- `当前设备已完成低强度验证`
- `后台保活正常`
- `已可以准备接入 Claude`

### 6.4 不满足条件时的处理

#### 设备未连接

提示：

- `请先连接玩具，再继续接入 Claude。`

动作：

- `去连接设备`

#### 适配器未绑定

提示：

- `请先为当前玩具选择适配器模板。`

动作：

- `去设备管理`

#### 验证未完成

提示：

- `请先完成低强度验证。验证通过前，Claude 还不能控制玩具。`

动作：

- `去开始验证`

## 7. 第二步：桥接准备

### 7.1 目标

让用户知道：Claude 不是直接连接蓝牙，而是先连接 ToyLink 的桥接能力。

### 7.2 页面应展示的信息

- Bridge 当前是否已就绪
- 当前 connector URL
- 当前 token 是否可用
- 支持的工具列表摘要

### 7.3 页面文案重点

这一步不要只展示技术字段，还要解释：

- `Claude 会通过这个连接请求 ToyLink 控制玩具。`
- `聊天和记忆仍然保留在 Claude。`
- `ToyLink 只负责安全地执行控制。`

### 7.4 页面操作

建议提供：

- `复制连接地址`
- `复制 token`
- `查看支持的控制工具`
- `下一步：去 Claude 添加`

### 7.5 安全提醒

这一步应明确提示：

- `不要把连接信息随意分享给他人。`
- `即使接入成功，微电流等敏感控制仍受本地安全限制。`

## 8. 第三步：Claude 添加 connector

### 8.1 目标

指导用户在 `claude.ai` 中完成一次 connector 配置。

### 8.2 页面结构建议

建议拆成“操作步骤卡片”，每张卡片只做一件事。

推荐结构：

1. 打开 `claude.ai`
2. 进入 connectors / integrations 管理位置
3. 添加新的 MCP connector
4. 填入 URL
5. 填入 token 或认证信息
6. 保存

### 8.3 页面展示内容

每一步建议同时展示：

- 中文步骤标题
- 一句简短解释
- 当前要复制的字段
- 一个“已完成，下一步”按钮

### 8.4 页面语气要求

必须避免写成工程手册。

推荐语气：

- `现在去 Claude 里添加一次连接。`
- `这一步只需要做一次，后面通常不需要重复。`
- `配置完成后，请回到你原来的对话继续测试。`

### 8.5 页面不应该做的事情

向导页不应伪装成自动完成 Claude 配置。

如果当前做不到自动化，就应该诚实地告诉用户：

- `接下来需要你在 Claude 中手动完成这一步。`

## 9. 第四步：原对话验证

### 9.1 目标

让用户回到原来的 Claude 对话里，确认那个已有记忆和关系的 Claude 已经获得控制能力。

### 9.2 推荐测试方式

这一步不建议让用户一上来就测试复杂命令。

推荐引导用户先说类似：

- `你现在可以控制我的设备了吗？`
- `先轻一点试试。`
- `先只开一点点震动。`

### 9.3 为什么这样设计

这样做有 3 个好处：

1. 测试风险低
2. 用户更容易判断是否已经接通
3. Claude 工具调用一旦失败，更容易定位问题

### 9.4 成功判定

系统层面不一定能直接知道 Claude 侧是否“理解了关系语境”，但至少可以通过以下信号判断技术接入是否成功：

- Bridge 收到来自 Claude 的工具调用
- App 本地成功执行
- 用户确认玩具有反应

## 10. 第五步：完成与故障排查

### 10.1 成功完成页

当首次接入成功后，页面应展示：

- `Claude 接入已完成`
- `现在请回到你原来的对话继续互动`
- `如果需要，也可以随时回到这里查看连接状态`

建议动作：

- `返回首页`
- `查看桥接状态`
- `查看安全说明`

### 10.2 失败时的排查页

如果用户在接入过程中失败，页面应按失败类型给出不同指导。

## 11. 失败场景与回退策略

### 11.1 本地未就绪

表现：

- 设备未连接
- 未完成验证
- Bridge 未启动

提示重点：

- 明确告诉用户“问题在 ToyLink 本地，不在 Claude”

回退动作：

- `去连接设备`
- `去完成验证`
- `去启动桥接`

### 11.2 Claude 端配置不完整

表现：

- 用户没有填 URL
- 用户没有填 token
- 用户保存失败

提示重点：

- 明确告诉用户“这一步要在 Claude 中完成”

回退动作：

- `重新查看接入步骤`
- `重新复制接入信息`

### 11.3 Connector 已添加但没有反应

表现：

- 用户说话了
- 但玩具没有反应
- Bridge 没收到调用，或本地没执行成功

提示重点：

- 区分是“Claude 没调用工具”还是“调用了但本地执行失败”

回退动作：

- `查看最近一次调用状态`
- `查看桥接状态`
- `重新测试一句低风险话术`

### 11.4 设备执行失败

表现：

- Claude 已调用工具
- 但本地返回错误

提示重点：

- 优先显示本地错误原因：
  - 设备断开
  - 未验证
  - EMS 超限
  - 后台保活失效

回退动作：

- `去桥接状态页`
- `去控制页`
- `去重新验证`

## 12. 页面状态模型

向导页建议至少定义以下状态：

- `loading`
- `blockedByLocalPrerequisite`
- `bridgePreparing`
- `bridgeReady`
- `awaitingClaudeSetup`
- `awaitingConversationTest`
- `success`
- `failure`

## 13. 与页面和路由的关系

### 13.1 推荐路由

建议新增独立页面路由，例如：

- `/claude-onboarding`

### 13.2 推荐入口来源

建议以下页面可进入：

- 验证完成页
- 首页
- MCP / Bridge 状态页
- 设置页中的 Claude 接入项

### 13.3 不建议放在什么位置

不建议把它埋在：

- 深层设置项里
- 仅开发者模式里
- 聊天页里

因为它是首发主流程，不是附属功能。

## 14. 与状态管理的关系

向导页不应自行维护系统真相状态。

它应该消费现有能力型 provider，例如：

- 活跃设备状态
- 适配器验证状态
- 桥接状态
- 前台服务状态

如果确实需要页面级 controller，建议只承担：

- 当前步骤索引
- 用户是否已完成某步确认
- 最近一次教程步骤复制状态

## 15. 当前不做的增强项

首发阶段不建议在这份向导里提前做：

- 自动导入 Claude 配置
- 多平台接入切换
- 视频式教程
- 一键云端诊断
- 复杂 FAQ 系统

这些都可以后续再补。

## 16. 验收标准

这份向导设计完成后，至少应满足：

1. 开发者知道应该做几步流程
2. 用户在每一步知道自己现在处于哪里
3. 用户知道下一步去哪
4. 失败时不会陷入“我不知道问题在哪”
5. 教程设计不破坏“保留 Claude 原对话”的核心目标
