# 官方支持模板库计划

## 1. 文档目的

这份文档把“官方支持模板库”正式纳入 ToyLink AI 的后续工作范围。

它要解决的不是协议实现问题本身，而是一个更现实的产品问题：

普通用户拿到 App 后，如果没有一批我们自己维护、自己验证过的模板，就很难独立完成：

1. 连接设备
2. 选择正确模板
3. 完成本机验证
4. 安全地交给 AI 通过 MCP 控制

所以，官方支持模板库不是可有可无的锦上添花，而是 ToyLink AI 面向普通用户落地时必须承担的一部分产品责任。

## 2. 目标定位

ToyLink AI 后续对设备支持分成三层：

### 2.1 官方支持

- 由项目维护者提供和维护
- 至少经过一轮真机低强度验证
- 面向普通用户优先展示和优先推荐
- 文案、推荐理由、验证步骤都应更完整

### 2.2 社区支持

- 来自用户导入或社区分享
- 允许导入和使用
- 但必须重新在本机验证
- 默认不直接视为可信模板

### 2.3 实验性支持

- 面向高级用户或开发者
- 允许手动创建、修改、调试模板
- 默认附带更明显的风险提示
- 不作为普通用户默认入口

## 3. 为什么必须做

如果我们不做官方支持模板库，普通用户会卡在这里：

- 不知道该选哪个模板
- 不知道导入别人的模板是否可信
- 验证失败后不知道应该换模板、重连设备还是放弃
- 看不懂高级参数，也不应该被要求看懂高级参数

这意味着：

- 纯靠“社区分享 + 用户自己试”不够
- 纯靠“高级模式手填参数”更不够

我们必须提供一条更傻瓜的主路径：

`扫描设备 -> 系统推荐官方模板 -> 本机验证 -> 启用 AI 控制`

## 4. 与当前架构的关系

官方支持模板库必须建立在现有架构之上，而不是另起一套系统。

必须保持这些设计不变：

- 官方模板和导入模板共用同一套 `AdapterManifest`
- 本机验证记录继续独立于模板文件存储
- `SafetyGuard` 继续统一拦截 MCP 和手动控制
- `stop_all()` 继续保持最高优先级
- 模板来源不会改变 `ToyDevice`、`Application`、`MCP` 的既有边界

也就是说，官方模板库是“模板来源和产品体验层的增强”，不是推翻当前架构。

## 5. 官方模板进入库的标准

一份模板要进入 `官方支持`，至少要满足：

1. `AdapterManifest` 结构通过静态校验
2. 设备匹配规则已经确认
3. 低强度验证向导可以完整跑通
4. `stop_all()` 已确认可靠
5. EMS 范围和安全提示已经核对
6. MCP 控制闸门可以正确识别该模板的验证状态
7. 导入、绑定、撤销验证、重新验证流程都能正常工作

## 6. 第一批模板选择原则

第一批不追求覆盖所有设备，而是优先覆盖：

- 当前 MVP 已经围绕其实现的设备或协议族
- 市场上更常见、更容易获取的设备
- 能复用现有 `codecKey` 或现有协议实现的设备

换句话说，前期要追求的是：

- 少量
- 高质量
- 可验证
- 容易推荐

而不是“模板数量越多越好”。

## 7. 后续实现任务

这件事进入正式开发时，建议拆成以下几个任务：

### Task 1: 模板来源建模

目标：
- 给模板增加来源类型，例如 `official / community / experimental`
- 为 UI 提供来源标识和筛选能力

完成标准：
- 系统能识别模板来源
- UI 能展示来源标签

### Task 2: 官方模板推荐规则

目标：
- 根据设备名、GATT 匹配信息和模板优先级给出推荐结果
- 面向普通用户优先展示官方模板

完成标准：
- 连接设备后可看到推荐模板
- 每个推荐项都有简短中文理由

### Task 3: 模板库展示与筛选

目标：
- 在设备管理或绑定流程中清楚区分三类模板
- 默认展示官方模板，社区和实验性模板次级展示

完成标准：
- 用户能一眼看懂哪些是推荐的、哪些是导入的、哪些风险更高

### Task 4: 官方模板最小回归清单

目标：
- 给每个官方模板建立最小真机验证清单
- 模板升级后可重复验证

完成标准：
- 至少包括连接、低强度验证、`stop_all()`、MCP 基本调用

### Task 5: 官方模板版本升级与重新验证

目标：
- 当官方模板升级时，系统能明确提示是否需要重新验证

完成标准：
- 版本变化、匹配规则变化或安全策略收紧时，状态转为 `needsReverify`

## 8. 放入项目后续工作的方式

这件事不放在“最前面立即做”，而放在“主链路闭合后、面向普通用户可用性提升”阶段推进。

建议顺序是：

1. 继续完成当前主链路闭环
2. 稳定验证、绑定、MCP 闸门和手动控制体验
3. 再补官方模板推荐和官方模板库建设

这样做的原因是：

- 先把底层链路做稳
- 再把普通用户入口做顺
- 最后扩展模板覆盖面

## 9. 当前结论

从现在开始，团队对这件事的统一结论应当是：

- 官方支持模板库是后续正式工作的一部分
- 它属于产品可用性建设，不属于“可做可不做”
- 它必须建立在现有适配器与验证架构之上
- 它的目标是降低普通用户的选择和验证门槛

后续再谈“支持更多设备”时，应优先问：

1. 这是新增官方模板？
2. 这是吸收社区模板？
3. 还是只是实验性支持？

只有先把这三类分清楚，产品和工程边界才会始终稳定。
