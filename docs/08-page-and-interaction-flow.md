# 页面与交互流设计

## 文档目的

这份文档定义 ToyLink AI MVP 的页面结构、页面职责和关键交互流。

它主要解决三个问题：

- 每个页面到底负责什么
- 页面之间如何跳转和协作
- 页面在不同状态下应该怎样表现

这份文档不是视觉稿，也不是组件清单，而是面向开发前设计的页面蓝图。

如果你是初级开发者，可以把它理解成：

“先把产品怎么走、每个页面做什么想清楚，再开始写页面代码。”

## 页面设计总原则

ToyLink AI 的页面设计必须遵守以下原则：

- 首页负责总览和主入口，不负责底层控制实现
- 页面只负责交互和展示，不负责协议细节
- MCP 是核心能力，不是隐藏功能
- 安全相关状态必须在 UI 上明确可见
- 页面状态设计要和领域状态机一致
- 页面出现错误时，要能通过 `Failure` 统一解释

一句话总结：

页面负责“让用户看懂和操作”，不负责“偷偷完成所有业务逻辑”。

## 路由与入口关系

MVP 阶段建议采用以下主路由：

- `/home`
- `/scan`
- `/control`
- `/mcp`
- `/chat`
- `/settings`
- `/lock`

推荐默认启动入口：

- 正常情况下进入 `/home`
- 若启用 App Lock 且当前需要认证，则先进入 `/lock`

## 全局导航结构

MVP 建议采用“首页总入口 + 次级页面跳转”的结构。

### 首页是主入口

首页承担以下职责：

- 当前设备和 MCP 服务状态总览
- 一键启动入口
- 快速导航中心
- 用户判断“当前系统是否可用”的第一观察面板

### 次级页面承担专项能力

- 扫描连接页：专注设备发现和连接
- 控制页：专注手动控制
- MCP 页：专注 MCP 服务状态与说明
- 聊天页：专注消息展示和工具调用记录
- 设置页：专注安全、偏好和隐私配置
- 锁页：专注解锁和访问控制

## 页面一览

MVP 页面范围如下：

1. 首页 `Home`
2. 设备扫描与连接页
3. 手动控制页
4. MCP 服务页
5. 聊天壳层页
6. 设置页
7. 应用锁页或解锁覆盖层

下面每个页面都必须明确：

- 页面目标
- 页面主要模块
- 关键状态
- 关键跳转
- 页面不该承担的职责

## 1. 首页 Home

## 页面目标

首页是 MVP 主入口页。

它的目标不是完成所有操作，而是让用户快速知道：

- 当前有没有活跃设备
- MCP 服务是否已启动
- 现在是否可以开始控制
- 该去哪个入口完成下一步

## 页面主要模块

首页建议包含以下模块：

- 环境状态区域
  - 蓝牙状态
  - 权限状态
  - App Lock 状态摘要
- 当前设备状态卡片
  - 是否已连接
  - 当前活跃设备名称
  - 最近状态摘要
- MCP 服务主卡片
  - 是否已启动
  - 本地端口或服务状态
  - 是否有活跃设备可供路由
- 一键启动主按钮
- 快速导航入口
  - 扫描连接
  - 手动控制
  - 聊天
  - 设置

## 关键状态

- `initial`
- `checkingEnvironment`
- `noDevice`
- `deviceReady`
- `mcpStopped`
- `mcpRunning`
- `quickStartInProgress`
- `error`

说明：

首页不是只有一个状态，而是一个“组合状态页”。例如：

- 设备已连接，但 MCP 未启动
- MCP 已启动，但没有活跃设备
- 环境检查失败，但用户仍可进入设置页修复问题

## 关键跳转

- 点击扫描连接 -> `/scan`
- 点击手动控制 -> `/control`
- 点击 MCP 主卡片 -> `/mcp`
- 点击聊天 -> `/chat`
- 点击设置 -> `/settings`
- 若检测到需要解锁 -> `/lock`

## 页面不该承担的职责

- 不直接发 BLE 命令
- 不直接启动具体 MCP tool
- 不直接操作底层存储
- 不直接处理协议编码

## 为什么这样设计

首页应该像“控制台”，而不是“工具箱内部零件本身”。

如果首页承担太多实际业务逻辑，后面会变成最难维护的页面。

## 2. 设备扫描与连接页

## 页面目标

这个页面专门负责：

- 请求必要权限
- 扫描候选设备
- 展示扫描结果
- 连接用户选中的设备

## 页面主要模块

- 扫描状态栏
- 权限与蓝牙环境提示区域
- 设备列表
- 重新扫描按钮
- 连接中的状态覆盖层或按钮 loading 态

## 关键状态

- `idle`
- `requestingPermissions`
- `bluetoothUnavailable`
- `scanning`
- `resultsAvailable`
- `connecting`
- `connected`
- `scanFailed`
- `connectionFailed`

## 关键跳转

- 从首页进入
- 连接成功后：
  - 可返回首页
  - 或继续进入控制页 `/control`

## 页面不该承担的职责

- 不保存复杂业务规则
- 不决定 EMS 安全策略
- 不决定 MCP 服务是否启动
- 不做协议级设备控制

## 为什么这样设计

扫描页只应该解决“找到并连上设备”这个问题。

如果它同时负责服务启动、手动控制、MCP 管理，用户会混乱，代码也会混乱。

## 3. 手动控制页

## 页面目标

手动控制页负责本地实时控制体验。

这是用户最直接感受到产品能力的页面。

## 页面主要模块

- 当前设备状态头部
- Suck 控制区域
- Vibe 控制区域
- EMS 控制区域
- 模式选择器
- 一键停止按钮
- 安全提示区域

## 关键状态

- `locked`
- `noActiveDevice`
- `ready`
- `applyingCommand`
- `awaitingEmsConfirmation`
- `stopping`
- `controlError`

## 关键跳转

- 无活跃设备时，引导回 `/scan`
- 被锁定时，引导到 `/lock`
- 成功控制后通常停留在当前页

## 页面不该承担的职责

- 不做协议编码
- 不直接写 BLE characteristic
- 不实现 MCP tool 行为
- 不单独定义安全规则

## 为什么这样设计

控制页必须“操作感强”，但业务边界必须稳。

它最容易让开发者为了快直接把 BLE 写进页面里，所以这里的边界要写得最死。

## 4. MCP 服务页

## 页面目标

这个页面负责让用户看懂 MCP 服务当前是什么状态，而不是让用户直接在页面里操作具体硬件协议。

## 页面主要模块

- 服务状态区域
- 地址或端口展示区域
- 当前活跃设备路由说明
- 启动或停止按钮
- 错误状态说明

## 关键状态

- `stopped`
- `starting`
- `runningWithoutDevice`
- `runningWithDevice`
- `error`

## 关键跳转

- 从首页 MCP 主卡片进入
- 若没有活跃设备，可引导去 `/scan`

## 页面不该承担的职责

- 不负责具体工具实现
- 不直接调用 BLE
- 不决定设备协议逻辑

## 为什么这样设计

MCP 是系统能力，不是一个“调试页小功能”。

但它也不应该变成“另一个控制页”。它更像服务面板。

## 5. 聊天壳层页

## 页面目标

聊天页负责展示消息、展示工具调用记录，并为未来模型接入预留用户体验外壳。

MVP 不要求它接入真实模型请求。

## 页面主要模块

- 消息列表
- 输入区域
- 工具调用记录消息卡片
- 系统状态提示

## 关键状态

- `empty`
- `messageReady`
- `toolEventVisible`
- `serviceUnavailable`
- `locked`

## 关键跳转

- 从首页进入
- 若锁定则进入 `/lock`
- 若没有可用设备或服务，可引导去 `/scan` 或 `/mcp`

## 页面不该承担的职责

- 不直接依赖某个模型厂商 SDK
- 不直接执行 BLE 控制
- 不单独维护一套与 MCP 脱节的工具调用逻辑

## 为什么这样设计

聊天页在 MVP 阶段是“体验壳层”，不是产品全部核心逻辑。

这样设计可以避免过早绑死 Claude 或某个中转服务。

## 6. 设置页

## 页面目标

设置页负责管理用户偏好、安全配置和隐私相关选项。

## 页面主要模块

- App Lock 设置
- EMS 安全提示与阈值说明
- 自动断开设置
- 自定义扫描前缀设置
- MCP 与隐私说明

## 关键状态

- `loading`
- `ready`
- `saving`
- `saveError`

## 关键跳转

- 从首页进入
- 某些安全项变更时可能要求临时解锁

## 页面不该承担的职责

- 不直接操作底层存储插件
- 不直接处理 MCP transport
- 不直接处理硬件控制

## 为什么这样设计

设置页很容易越写越杂，所以它必须只负责“配置”，不负责“执行底层行为”。

## 7. 应用锁页或解锁覆盖层

## 页面目标

应用锁页负责保护敏感页面和敏感操作。

## 页面主要模块

- 解锁说明
- 生物识别触发入口
- 失败提示
- 返回或取消入口

## 关键状态

- `lockRequired`
- `authenticating`
- `unlocked`
- `lockedOut`

## 关键跳转

- 从任意敏感页面前置进入
- 解锁成功后返回原目标页

## 页面不该承担的职责

- 不替代业务安全规则
- 不决定 EMS 是否允许执行
- 不跳过 `SafetyGuard`

## 为什么这样设计

App Lock 是访问控制，不是业务安全的替代品。

就算解锁成功，EMS 软上限和 MCP 校验规则也仍然要继续生效。

## 异常状态和空状态策略

所有页面都应明确两类状态：

- 空状态
- 错误状态

推荐原则：

- 没有活跃设备时，不显示误导性的“可控制状态”
- 没有权限时，不假装只是“没扫描到设备”
- MCP 没启动时，不假装聊天可用
- 锁定状态要优先于普通展示状态

## 与核心对象的关系

### 与 `Failure` 的关系

页面错误态必须建立在 `Failure` 上，而不是每个页面自己发明一套错误解释。

### 与 `ActiveDeviceRegistry` 的关系

页面不自己维护“谁是当前设备”的真相源，统一通过 `ActiveDeviceRegistry` 获取。

### 与 `SafetyGuard` 的关系

页面可以发起操作，但不能替代 `SafetyGuard` 做最终安全判断。

### 与 `McpToolRouter` 的关系

MCP UI 只展示服务和状态，不模拟或重写 `McpToolRouter` 的逻辑。

## 关键交互流

### 1. 首次打开 App

流程：

1. 打开 App
2. 若启用 App Lock 且需要验证，进入 `/lock`
3. 解锁后进入 `/home`
4. 首页进行环境检查
5. 用户进入 `/scan`
6. 请求权限并开始扫描

### 2. 首页一键启动

流程：

1. 用户点击一键启动
2. 系统检查权限和蓝牙状态
3. 系统尝试连接最近一次设备或引导用户选择设备
4. 连接成功后启动前台服务
5. 启动 MCP 服务
6. 首页进入可控制状态

### 3. 首页进入控制页

流程：

1. 用户点击手动控制
2. 若无活跃设备，提示并引导去扫描页
3. 若已锁定，进入锁页
4. 若设备可用，进入控制页
5. 调节强度后触发命令
6. 状态更新回流到页面

### 4. EMS 超过软上限

流程：

1. 用户调整 EMS 到大于 `8`
2. 页面进入 `awaitingEmsConfirmation`
3. 显示明确警示
4. 用户取消则恢复到安全状态
5. 用户确认后再继续交给应用层处理

说明：

页面可以负责展示确认，但最终安全判断仍然应由 `SafetyGuard` 统一执行。

### 5. MCP 已运行但设备断开

流程：

1. MCP 服务继续保持 `running`
2. 首页显示服务在运行，但当前无可用活跃设备
3. MCP 页面显示 `runningWithoutDevice`
4. 工具调用返回结构化错误

### 6. App 回到前台

流程：

1. App 从后台恢复
2. 若启用 App Lock 且超出锁定条件，进入 `/lock`
3. 解锁成功后返回原页面

### 7. 无活跃设备时进入聊天页

流程：

1. 用户进入 `/chat`
2. 页面可以展示聊天壳层
3. 若服务或设备不可用，显示引导信息
4. 引导用户前往 `/scan` 或 `/mcp`

## MVP 最小用户主路径

推荐用以下路径判断页面设计是否成型：

1. 用户打开 App
2. 进入首页
3. 去扫描页连接设备
4. 返回首页看到设备状态已就绪
5. 通过首页进入控制页
6. 完成一次手动控制
7. 返回首页启动 MCP
8. 查看 MCP 状态页确认服务在运行
9. 进入聊天页查看工具调用壳层
10. 进入设置页管理安全和偏好

## 页面边界总结

为了后续开发不走偏，可以把页面边界记成下面这几句：

- 首页负责总览，不负责底层执行
- 扫描页负责连接，不负责控制
- 控制页负责交互，不负责协议
- MCP 页负责服务状态，不负责工具实现
- 聊天页负责体验壳层，不负责模型耦合
- 设置页负责配置，不负责底层存储实现
- 锁页负责访问控制，不替代业务安全

## 给初级开发者的建议

当你在写页面时，如果拿不准某段逻辑该不该放进页面，可以这样问自己：

1. 这段逻辑是在“展示和交互”吗？
2. 这段逻辑是不是在决定协议、BLE、MCP 或安全规则？

如果答案是第二种，那它大概率就不该直接写在页面里。

这样做虽然一开始看起来慢一点，但后面你会轻松很多。
