# 实施路线图

## 1. 文档目的

这份文档不是讲架构原理，而是把 ToyLink AI 的开发顺序拆成可以直接照着做的实施路线。

当前路线已经从“单一 SOSEXY MVP”收束为更清晰的第一阶段目标：

- 用户在 App 内自行配对玩具
- 用户通过模板或适配器文件接入设备
- 用户先完成低强度验证
- 外部 AI 通过本地 MCP 控制已验证设备

## 2. 开发总原则

- 先定规则，再写代码
- 先打通主闭环，再补增强能力
- 先把安全边界做稳，再扩设备兼容
- UI 不直接依赖 BLE、MCP 或存储插件
- 任何控制命令都必须经过 `SafetyGuard`

## 3. 分阶段顺序

第一阶段推荐按以下 7 个阶段推进。

1. `Phase 1: Core + Domain 基础层`
2. `Phase 2: Application 编排层`
3. `Phase 3: Adapter System 基础能力`
4. `Phase 4: BLE 连接与设备验证闭环`
5. `Phase 5: MCP 控制闭环`
6. `Phase 6: 设置、安全与分享体验`
7. `Phase 7: 质量收尾与演示准备`

## 4. Phase 1: Core + Domain 基础层

### 目标

建立全项目稳定边界，让后续功能不再依赖“边写边定规则”。

### 要产出的能力

- `Failure` 统一抽象
- 核心实体：`ToyDevice`、`DeviceStatus`、`ControlCommand`、`SafetyPolicy`
- 主题、路由、常量基础
- Clean Architecture 与 Feature-First 目录稳定下来

### 完成标准

- 项目不再残留默认 Flutter 模板思路
- 领域接口稳定，可支撑后续 adapter、BLE、MCP
- UI、application、infrastructure 的边界清楚

### 常见误区

- 一上来先写页面
- 在 domain 中混入插件类型
- 没定 `Failure` 就四处抛异常

## 5. Phase 2: Application 编排层

### 目标

把控制、验证和状态同步的主逻辑集中起来，让 UI 与 MCP 共用同一条控制链。

### 要产出的能力

- `ActiveDeviceRegistry`
- `SafetyGuard`
- 基础 use case
- Riverpod 应用层 provider 骨架

### 完成标准

- UI 与 MCP 都不直接碰 BLE
- EMS 安全策略有统一入口
- 活跃设备路由逻辑稳定

### 常见误区

- 把所有逻辑塞进 page controller
- 在 use case 里直接写 BLE 细节
- 绕过 `SafetyGuard`

## 6. Phase 3: Adapter System 基础能力

### 目标

让“模板/适配器文件 -> 本地静态校验 -> 本地验证状态”这条链路成型。

### 要产出的能力

- `AdapterManifest`
- `VerifiedAdapterRecord`
- `AdapterRegistry`
- `AdapterValidator`
- `VerifiedAdapterStore`
- 内置 `codecKey` 注册表

### 完成标准

- 适配器文件和本地验证状态彻底分离
- 内置模板可被选择和静态校验
- 未验证适配器无法进入控制链

### 常见误区

- 把 `verified` 写进分享文件
- 让适配器能放宽系统安全上限
- 只靠设备名前缀做最终匹配

## 7. Phase 4: BLE 连接与设备验证闭环

### 目标

完成“扫描 -> 连接 -> 选择模板 -> 低强度验证 -> 启用设备”的主闭环。

### 要产出的能力

- BLE 扫描与连接
- GATT 发现与写入队列
- 模板匹配与精匹配
- 验证向导
- `stop_all` 抢占行为

### 完成标准

- 用户可以在 App 里自行配对玩具
- 用户可以完成一轮低强度验证
- 设备启用状态与验证状态联动正常

### 常见误区

- 只实现连接，不实现验证闭环
- `stop_all` 只是普通命令
- 失败后没有清晰回退路径

## 8. Phase 5: MCP 控制闭环

### 目标

让外部 AI 可以通过本地 MCP 安全控制已验证设备。

### 要产出的能力

- MCP 启停与状态展示
- 工具注册与路由
- `adapterNotVerified` 拒绝逻辑
- EMS 软上限错误返回

### 完成标准

- `set_suck`
- `set_vibe`
- `set_ems`
- `set_all`
- `stop_all`
- `get_status`

以上工具都能稳定路由到活跃设备，并遵守本地安全规则。

### 常见误区

- MCP 直接调用 BLE
- MCP 与 UI 使用不同的安全逻辑
- 未验证设备也允许 MCP 调用

## 9. Phase 6: 设置、安全与分享体验

### 目标

补齐用户真正可用的体验层，而不只是工程闭环。

### 要产出的能力

- 适配器导入/导出
- 分享文件说明与兼容提示
- App Lock
- 隐私设置
- 设备管理页

### 完成标准

- 用户可导出分享适配器文件
- 导入分享文件后会重新验证
- 敏感页面受锁屏保护

### 常见误区

- 分享了验证结果而不是分享 manifest
- 只做 UI 锁，不拦实际敏感操作
- 设置页直接操作插件

## 10. Phase 7: 质量收尾与演示准备

### 目标

把项目从“能跑”提升到“可演示、可维护、可继续迭代”。

### 要产出的能力

- 单元测试补齐
- Widget 测试补齐
- 关键集成测试
- 日志脱敏核查
- 文档与代码一致性检查

### 完成标准

- 核心闭环可演示
- 关键失败路径有统一表现
- 所有关键安全规则都可测试

### 常见误区

- 只测成功路径
- 功能能跑就算完成
- 文档和代码各走各的

## 11. MVP 最小验收路径

第一阶段最小验收应是：

1. 用户打开 App
2. 允许蓝牙相关权限
3. 扫描到目标设备
4. 成功连接
5. 选择内置模板或导入适配器文件
6. 完成低强度验证
7. 启动本地 MCP
8. 外部 AI 成功调用 `get_status`
9. 外部 AI 成功调用 `set_suck` 或 `set_vibe`
10. `stop_all` 可立即生效
11. 超过 EMS 软上限的 MCP 请求被拒绝

## 12. 给初级开发者的建议

如果不知道现在该做什么，就看两件事：

- 当前处于哪个阶段
- 当前阶段的完成标准是什么

不要同时跨多个阶段乱写。一步一步做，反而更快、更稳。
