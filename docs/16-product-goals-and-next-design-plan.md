# 产品目标与下一阶段设计计划

## 1. 文档定位

本文档用于正式覆盖早期“本地聊天壳层 + 本地 MCP 优先”的产品目标描述。

最新推进准绳见 `docs/22-byo-ai-hardware-connector-roadmap.md`。若本文档与该准绳冲突，以 `docs/22` 为准。

当前项目已经完成了一轮更明确的产品收敛，因此本文件现在作为：

- 当前产品目标的正式定义
- 后续设计与开发排期的总入口
- 与 `CODEX.md`、开发守则、专题设计文档对齐的产品级基线

保留的有效内容包括：

- Clean Architecture 与 Feature-First 约束
- `ToyDevice`、`SafetyGuard`、`Failure` 等技术守则
- 适配器绑定、验证、安全拦截的底层方向

被覆盖的旧前提包括：

- “内置聊天是首发主功能”
- “首发优先本地 MCP 直连外部 AI”
- “先做多平台 AI 接入”

## 2. 当前正式产品定义

ToyLink AI 当前阶段的正确产品定义是：

**BYO-AI Hardware Connector：用户继续留在自己原有的 AI 聊天 Web/App 中互动，ToyLink 负责把该 AI 的工具调用能力安全连接到用户手机已连接的硬件。**

这句话有 5 个关键点：

1. 用户的主对话发生在用户自己的 AI 环境中，不发生在 ToyLink 内。
2. AI 的记忆、语气、角色关系保留在原平台侧，不迁移到 ToyLink。
3. ToyLink 不是聊天替代品，而是现实互动桥接器。
4. ToyLink 的核心职责是硬件连接、安全验证、控制执行和 connector / tool 暴露。
5. Claude 是第一条具体接入路径；GPT / ChatGPT、自建 Agent 等属于后续接入矩阵。

## 3. 当前首发范围

### 3.1 首发必须完成的能力

安全 V0 必须围绕 BYO-AI 主链路打通以下闭环：

1. 手机连接玩具
2. 绑定适配器
3. 完成低强度验证
4. 启动桥接能力
5. 提供 AI Connector 接入信息与教程
6. 允许用户原有 AI 环境调用 `get_status` 和 `stop_all`

安全 V0 不承诺远程刺激控制。`set_suck`、`set_vibe`、`set_ems`、`set_all` 必须等后续安全门禁完成后再开放。

### 3.2 首发明确不做的能力

为了避免产品目标再次发散，首发阶段明确不做：

- 不做内置聊天主功能
- 不做“记忆迁移”
- 不做“关系档案重建”
- 不做 GPT / 酒馆 / 自建终端的正式发布兼容；但 GPT / ChatGPT 必须进入后续接入矩阵设计
- 不做“所有平台都能一键接入”的产品承诺

如果后续扩展这些方向，必须新开设计文档或 ADR，不得默认沿用旧假设。

## 4. 当前产品为什么这样定义

这个定义不是凭空拍脑袋决定的，而是基于用户目标和平台边界共同收敛出来的。

### 4.1 用户目标决定了不能换聊天容器

用户想要的是：

- 继续在自己原有的 AI 聊天环境里互动
- 保留原来的记忆、互动方式和关系感
- 在此基础上获得更多现实互动

这意味着一旦要求用户切到 ToyLink 内重新开聊，产品核心价值就被破坏了。

### 4.2 平台边界决定了必须走 Remote MCP

当前第一条目标路径是 Claude Remote MCP，但产品边界不应绑定到 Claude。

这意味着：

- 本地 App 不能作为最终聊天容器
- 手机本地 MCP 不是最终主路线
- 首发接入方式必须围绕 Remote Connector / Tool 调用设计
- MCP 是首选协议，但不是产品定义本身

### 4.3 教程案例证明路线可行

已有网页级教程证明了如下链路可运行：

`用户已有 AI -> Remote Connector / Tool -> ToyLink Bridge -> 手机本地执行层 -> BLE 玩具`

这说明我们的方向不是概念验证，而是产品化改造：

- 去掉 Termux 依赖
- 去掉 HTML/Chrome 前台依赖
- 用 App 接管 BLE、状态和安全
- 用更清晰的接入教程服务普通用户

## 5. 当前系统职责边界

### 5.1 用户自有 AI 的职责

用户自有 AI 负责：

- 保持原对话
- 维持角色关系和记忆
- 在互动中决定是否调用工具

用户自有 AI 不负责：

- BLE 连接
- 玩具协议
- 安全上限
- 本地适配器验证

### 5.2 ToyLink 的职责

ToyLink 负责：

- 扫描并连接玩具
- 管理适配器
- 执行低强度验证
- 管理本地安全策略
- 执行工具调用
- 把控制能力暴露给 Claude

ToyLink 不负责：

- 替代 Claude 进行聊天
- 复制 Claude 的记忆体系
- 让用户在新会话里重新培养关系

## 6. 首发用户主流程

首发版本的推荐主流程如下：

1. 用户安装 ToyLink
2. 用户在 App 中连接玩具
3. 用户为当前玩具绑定适配器
4. 用户完成低强度验证
5. 用户进入 Claude 接入向导
6. 用户按照对应平台教程，在自己的 AI Web/App 中添加一次 connector / tool
7. 用户回到原来的 AI 对话继续聊天
8. AI 在原对话中调用 ToyLink 暴露的安全工具

这个流程的关键不是“聊天”，而是“首次接入教程”。

## 7. 当前页面优先级

由于产品主线已经收敛，当前页面优先级也需要同步调整。

### 7.1 当前主页面

首发阶段的核心页面应为：

- 首页
- 扫描连接页
- 手动控制页
- 设备管理页
- 验证页
- MCP / 桥接状态页
- AI Connector 接入教程页
- 设置页

### 7.2 当前非主页面

以下页面不再视为首发主功能页面：

- 聊天页

聊天页如果保留，当前只应作为：

- 调试页
- 本地工具调用测试页
- 后续实验能力入口

不得再作为首发产品主路径。

## 8. 当前最重要的设计任务

在继续大规模编码前，最值得补齐的是以下设计任务。

### 8.1 BYO-AI Connector 产品定义定稿

需要把以下问题正式写死：

- 产品一句话定义
- 首发支持范围
- 首发不做项
- 用户主路径
- 教程入口位置

### 8.2 Remote Connector 架构设计

需要明确：

- 用户 AI、Remote Connector、ToyLink App、玩具之间的链路
- 哪些请求走远程
- 哪些执行必须留在本地
- 如何做状态展示和故障反馈

### 8.3 AI Connector 接入教程设计

需要明确：

- 教程何时出现
- 教程页面要展示哪些信息
- 用户需要复制或填写哪些内容
- 如何验证“接入已成功”

### 8.4 后台保活与在线状态设计

需要继续强化：

- App 在后台时的连接稳定性
- 桥接链路断开后的提示策略
- 前台服务与用户可见状态的一致性

## 9. 当前首发风险

当前路线最大的风险不是 UI，而是接入闭环和稳定性。

### 9.1 Remote Connector 接入复杂度

首次 connector 配置会是用户门槛。

解决方向：

- 做非常清晰的中文教程
- 提供尽量可复制的信息
- 在 App 内做状态检查和错误提示

### 9.2 后台稳定性

Claude 发出控制请求时，ToyLink 必须在线、可执行、可恢复。

解决方向：

- 前台服务
- 后台保活检查
- 明确的连接状态 UI

### 9.3 安全边界

由于 Claude 会在原对话中自行判断何时调用工具，安全限制必须非常硬。

解决方向：

- 保持 `SafetyGuard` 为唯一控制前置
- EMS 默认强限制不变
- 保留 `stop_all()` 的最高优先级

## 10. 当前默认开发顺序

从现在开始，建议按以下顺序推进产品实现：

1. 完成产品定义与架构文档收敛
2. 完成 Claude 接入教程设计
3. 打通 Remote MCP 主链路
4. 强化后台保活与状态反馈
5. 完善错误提示与故障排查
6. 最后再做视觉打磨和次级能力

## 11. 当前结论

当前项目不应该再被定义成：

- 一个内置 AI 聊天 App
- 一个本地玩具控制器
- 一个一开始就面向所有 AI 平台的万能桥

当前项目应该被定义成：

**一个让用户已有 AI 聊天环境获得安全硬件连接能力的 BYO-AI Hardware Connector。**

这就是现在最值得坚持的产品主线。
