# 产品目标与下一阶段设计计划

## 1. 当前产品目标

ToyLink AI 第一阶段的清晰目标是：

让普通用户在手机里完成玩具配对、模板选择和低强度验证，然后让外部 AI 通过本地 MCP 安全控制已验证设备。

这意味着产品重点是：

- 本地配对
- 本地验证
- 本地安全控制
- 本地 MCP 接口

而不是：

- 先做复杂聊天界面
- 先做云端转发
- 先做开放执行插件平台

## 2. 成功标准

若第一阶段目标完成，用户应能做到：

1. 在 App 内扫描并连接玩具
2. 选择内置模板或导入别人分享的适配器文件
3. 跑完验证向导并确认反应正确
4. 启用设备
5. 让外部 AI 客户端通过 MCP 控制设备

## 3. 当前设计结论

已经锁定的关键结论如下：

- 第一阶段外部接口只做 `MCP`
- 适配器模式采用 `说明书文件 + App 内置翻译器`
- 适配器文件允许分享
- 验证状态不允许分享复用
- 所有控制都必须经过 `SafetyGuard`
- `stop_all` 是最高优先级
- EMS 软上限规则在 UI 与 MCP 上分开处理

## 4. 接下来的设计顺序

为了进入编码前不再返工，建议按这个顺序收尾设计。

### 4.1 适配器系统定稿

对应文档：

- `docs/15-adapter-sdk-and-verification-workflow.md`

必须确认的内容：

- manifest 字段
- 模板种类
- 匹配规则
- 验证状态存储
- 分享与重验证规则

### 4.2 页面与表单定稿

重点是适配器创建与验证流程。

必须确认的内容：

- 普通模式用户填什么
- 高级模式用户填什么
- 导入分享文件后页面怎么走
- 验证失败如何回退

### 4.3 路由与 provider 同步

若产品主线从“内置聊天”转为“外部 AI + MCP”，则：

- 首页
- 扫描连接页
- 控制页
- MCP 页
- 设备管理页

应成为第一阶段主页面。

### 4.4 测试与错误码同步

必须确认：

- 适配器相关 Failure
- MCP 错误码映射
- 验证闭环的单元与集成测试

## 5. 编码前最终就绪条件

只有满足以下条件，才建议全面进入编码：

1. `CODEX.md` 与专题文档一致
2. 适配器规范已定稿
3. 主页面与主状态流已定稿
4. 适配器错误码已定稿
5. 第一阶段验收路径已定稿

## 6. 编码建议顺序

文档全部定稿后，建议按这个顺序进入实现：

1. 适配器领域模型与存储抽象
2. `AdapterRegistry` 与 `AdapterValidator`
3. 模板选择与导入 UI
4. BLE 精匹配与验证向导
5. MCP 与验证状态联动
6. 设置、分享与安全页
